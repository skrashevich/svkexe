Features Built for people who host their own things.
01
Persistent by design

Boxes live on ZFS datasets. Power cycle the host, reboot containers — your work is exactly where you left it. No ephemeral filesystem nonsense.

02
AI agent per box

Shelley runs inside each container on :9000. It has full filesystem access to that container and nothing else. Your context, your CPU, your data.

03
Ownership enforced at the edge

The gateway checks user→container mapping on every single proxy request. No amount of clever URL crafting gets you into someone else's box.

04
One-command backups

Incus snapshots + SQLite dump, scripted to run on a cron. Restore to any point in time. No proprietary backup format, no cloud dependency.

05
Ships as one Go binary

Gateway, dashboard, and SSH menu are the same process. Copy it to the server, drop a systemd unit, done. No node_modules, no Python runtime, no Docker-in-Docker.

06
Actually small

~100 boxes on 64 GB RAM at idle. Containers share the host kernel. You're paying for what you use, not what the orchestrator hoards.

FAQ Questions you'd probably ask first.
How is this different from a VPS? +
A VPS is a single VM you rent. svkexe lets you run many containers on hardware you own, with an AI coding agent pre-installed in each one and a shared control plane that handles routing, auth, and ownership. You get the multi-tenant ergonomics of a cloud platform without the cloud bill or the lock-in.
Does it require Kubernetes or Docker? +
No. svkexe uses Incus (the community fork of LXD) to manage LXC containers. There is no Docker daemon, no container registry pull, and definitely no Kubernetes. The runtime dependency is Incus + a ZFS pool. That's it.
What does "Shelley" do exactly? +
Shelley is a small Go process that runs inside each container on port :9000. It exposes a WebSocket interface for the AI coding agent, handles file reads/writes, and proxies terminal sessions. It is scoped entirely to that container's filesystem — it has no network reach outside the LXC boundary.
Can I bring my own LLM provider? +
Yes. Shelley reads its provider config from a tmpfs mount at a path unique to each container (invariant S5). You can configure OpenAI, Anthropic, Ollama, or any OpenAI-compatible endpoint. Keys never touch the gateway process and are never stored in SQLite.
Is there a hosted version? +
Not currently. The entire point of svkexe is that it runs on your server. There are no SaaS dependencies in the critical path — no auth cloud, no telemetry endpoint, no license server. If you want hosted, rent a bare-metal server and run this on it.
